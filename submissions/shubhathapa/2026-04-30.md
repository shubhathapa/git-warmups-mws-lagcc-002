# Questions to answer in your daily log

- In your own words, what is the "staging area" and why does it exist?
staging area is a middle step between my working directory and the commit history. it helps to choose exactly which changes I want to include in the next commit instead of saving everything at once. 

- What's the difference between `git add` and `git commit`?
git add puts changes into the staging area 
git commit permanently saves those staged changes into the Git history with a message.
- What does `git restore --staged <file>` do?
removes a file from the stagiing area but keeps the actual changes in the working directory. 


## Questions to answer in your daily log

- In your own words, what's a "stash"?
stash is a place to save current uncommitted changes to switch brances or work on something else without losing your work

- What's the difference between `git stash pop` and `git stash apply`?
git stash pop restores the stashed changes and removes them from the stash list 
git stash apply restores the canges but keeps tem saved in the stash so i can reuse them later

- When would you use a named stash (`stash push -m "message"`) vs an unnamed one?
A named stash is useful when there are multiple stashes and want to remember what each one id for. An unnamed stash is fine when i only have one or dont need to track it closely.

- What's the difference between a lightweight tag and an annotated tag?
a lightweight tag is a simple pointer to commit with no extra information.
an annotated tag includes additional details like the tagger's name, date and a message, making it more useful for important versions.

- Why might you use tags instead of branches for releases?
tags mark specific points in history and dont chanhe while branches can keep on moving. tags are better for stable versions.
